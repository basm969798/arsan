✅ منطق العمل النهائي (Final Business Logic)

1️⃣ أنواع الطلبات
يوجد نوعان فقط من الطلبات داخل النظام:
أ — طلب مباشر
التاجر يختار قطعة من الكتالوج.
يحدد مورد معيّن.
يرسل الطلب مباشرة.
المورد:
يقبل الطلب.
أو يرفض الطلب.
عند قبول المورد:
👉 يتم تثبيت السعر داخل الطلب.

ب — طلب نشر عام
التاجر ينشر طلب بدون اختيار مورد.
الموردون المؤهلون يرون الطلب.
يقدم الموردون عروض.
التاجر يختار عرض واحد فقط.
عند اختيار العرض:
👉 يتم تثبيت السعر داخل الطلب.

2️⃣ دورة حياة الطلب (التسلسل الأساسي)
التسلسل ثابت ولا يمكن تجاوزه:
1️⃣ إنشاء الطلب
2️⃣ انتظار رد المورد / استقبال العروض
3️⃣ قبول المورد أو اختيار عرض
4️⃣ تجهيز الطلب
5️⃣ استلام الطلب عبر رمز الاستجابة (QR)
6️⃣ إغلاق الطلب مالياً

3️⃣ تثبيت السعر
السعر يصبح ثابتاً عند:
👉 قبول المورد (في الطلب المباشر).
👉 اختيار العرض (في طلب النشر العام).
أي تغيير لاحق في أسعار المورد:
❌ لا يؤثر على الطلب الحالي.

4️⃣ الاستلام
الاستلام هو إجراء ذري واحد يشمل:
التحقق عبر رمز الاستجابة (QR).
تأكيد الاستلام.
انتقال الحالة إلى "مستلم".
فتح مرحلة الإغلاق المالي.

5️⃣ إغلاق الطلب (القرار المالي)
بعد الاستلام يجب اختيار أحد الخيارين:
✅ إغلاق نقدي.
✅ تسجيل دين.
هذا القرار:
👉 إلزامي قبل إنهاء الطلب.

6️⃣ السجل المالي
قاعدة ذهبية:
👉 لا يتم إنشاء سجل مالي قبل إكمال الطلب.
السجل المالي:
غير قابل للتعديل.
يعتمد على إضافة أحداث فقط (append-only).

7️⃣ الدفعات الجزئية
إذا تم اختيار الدين:
يمكن تسجيل دفعات متعددة.
كل دفعة تصبح سجل مالي مستقل.

8️⃣ نقل الملكية بين التجار
يمكن نقل قطعة من تاجر إلى تاجر:
1️⃣ إرسال طلب نقل.
2️⃣ قبول الطرف الثاني.
3️⃣ نقل الملكية.
4️⃣ تسجيل حدث مالي.

9️⃣ قواعد الإلغاء والرفض
إذا لم يرد المورد:
يمكن للتاجر إلغاء الطلب.
إذا رفض المورد:
يعود الطلب إلى حالة مفتوحة.
يمكن اختيار مورد آخر.
أو تحويله إلى نشر عام.
بعد قبول المورد:
يمكن الإلغاء قبل الاستلام فقط.
بعد الاستلام:
❌ لا يوجد إلغاء.

🔟 حدود النظام (قرارات ثابتة)
لا يوجد دفع إلكتروني.
لا يوجد توصيل داخل النظام.
المنصة لا تمسك الأموال.

✅ قواعد الاتساق والتزامن النهائية الموحدة 

🧱 ١️⃣ السيرفر هو مصدر الحقيقة الوحيد
جميع تغييرات الحالة تتم عبر السيرفر فقط.
الواجهة ترسل طلباً فقط.
السيرفر يتحقق ثم يقرر التغيير.

🔒 ٢️⃣ الانتقال المشروط للحالة
لا يتم تغيير الحالة مباشرة.
يتم التغيير فقط إذا كانت الحالة الحالية مطابقة للحالة المتوقعة.
إذا تغيرت الحالة قبل التنفيذ يتم رفض العملية.

⚡ ٣️⃣ القرار الذري (Atomic Operation)
كل عملية حساسة يجب أن تنفذ كوحدة واحدة تشمل:
التحقق.
اتخاذ القرار.
تحديث الحالة.
إما تنفذ بالكامل أو تفشل بالكامل.

🔁 ٤️⃣ منع التكرار (Idempotency)
كل عملية لها معرف فريد.
إعادة إرسال نفس الطلب لا تنفذه مرتين.
مهم لعمليات القبول، الإلغاء، والإغلاق المالي.

👥 ٥️⃣ أولوية العمليات
عند التعارض يتم تطبيق قواعد ثابتة:
قبول المورد يمنع تعديل الطلب.
بدء مرحلة لاحقة يمنع الرجوع لمرحلة سابقة.
أول انتقال صحيح زمنياً هو المعتمد.

🧠 ٦️⃣ فصل الحالات
الحالات مقسمة منطقياً:
حالة دورة الطلب.
الحالة المالية.
حالة الملكية.
حالة الكتالوج.
ولا يتم دمجها في حالة واحدة.

⏱️ ٧️⃣ رفض العمليات المتأخرة
كل عملية تتحقق من الحالة الحالية قبل التنفيذ.
العمليات التي وصلت متأخرة يتم تجاهلها.

📜 ٨️⃣ تسجيل الأحداث (Event Logging)
كل انتقال حالة يسجل:
الحالة السابقة.
الحالة الجديدة.
المستخدم المنفذ.
وقت التنفيذ.

🔄 ٩️⃣ إعادة المحاولة الآمنة
العمليات يمكن إعادة إرسالها بأمان.
النظام يمنع التنفيذ المكرر.

🧪 🔟 منع السباق (Race Condition Protection)
عند العمليات الحساسة يتم التحقق من الحالة والمخزون فورياً.
يسمح بتنفيذ عملية واحدة فقط عند التعارض.
العمليات الأخرى يتم رفضها تلقائياً.

💾 1️⃣1️⃣ تثبيت السعر
السعر يتم تثبيته عند قبول الطلب.
أي تعديل لاحق في سعر المورد لا يؤثر على الطلب الحالي.

🔔 1️⃣2️⃣ ترتيب الأحداث
ترتيب التنفيذ يعتمد على الحالة الحالية في السيرفر.
ليس على ترتيب وصول الإشعارات.

✅ طريقة البحث النهائية الموحدة 
🧱 ١️⃣ مبدأ البحث الأساسي
البحث هو نقطة الدخول الرئيسية للنظام.
يبدأ المستخدم مباشرة بالبحث.
لا توجد خطوات أو قوائم قبل البحث.

🔎 ٢️⃣ أنواع البحث
يوجد نوعان فقط:
✅ أ — البحث النصي المباشر
يمكن إدخال:
اسم القطعة
موديل السيارة
سنة الصنع
رقم القطعة
وصف عام
القواعد:
كل كلمة إضافية تضيق النتائج تدريجياً.
دعم الكلمات المرادفة.
تحمل أخطاء إملائية بسيطة.

✅ ب — البحث عبر رقم تعريف المركبة (VIN)
عند إدخال VIN:
1️⃣ البحث داخل البيانات المحلية أولاً.
2️⃣ إذا لم توجد نتيجة يتم استخدام API خارجي كـ fallback فقط.
3️⃣ جلب بيانات السيارة تلقائياً.
4️⃣ تعبئة معلومات السيارة داخل البحث تلقائياً كفلتر.
5️⃣ حفظ البيانات محلياً للاستخدام لاحقاً (cache).
ملاحظات:
VIN يستخدم لتضييق البحث فقط.
لا يتم اختيار القطعة تلقائياً اعتماداً على VIN.

📦 ٣️⃣ عرض النتائج
عرض القطعة أولاً وليس المورد.
تظهر القطعة كعنصر واحد داخل النتائج.
عند فتح القطعة يتم عرض قائمة الموردين والأسعار.

💰 ٤️⃣ ظهور الأسعار
الأسعار تظهر مباشرة داخل قائمة الموردين.
الهدف تسريع اتخاذ القرار.

📊 ٥️⃣ ترتيب النتائج
ترتيب القطع يكون حسب:
1️⃣ القطع المتوفرة فعلياً.
2️⃣ القطع التي لديها عدد موردين أكبر.
3️⃣ باقي النتائج.

⚡ ٦️⃣ البحث الفوري
النتائج تظهر أثناء الكتابة مباشرة.
لا يوجد زر بحث إجباري.

🔤 ٧️⃣ التعامل مع اختلاف الكتابة
دعم المرادفات.
تحمل أخطاء إملائية بسيطة.
توحيد النص قبل البحث (normalization).

🚫 ٨️⃣ إذا لم توجد نتائج
لا يظهر "لا توجد نتائج".
يظهر زر إنشاء طلب نشر عام مباشرة.
يتم استخدام بيانات البحث تلقائياً.

📌 ٩️⃣ قواعد أساسية
البحث يبدأ من الكتالوج التقني.
الموردون يُجلبون بعد اختيار القطعة.
API الخارجي يستخدم فقط عند الحاجة.
لا يوجد طريق مسدود.
البحث هو بداية الطلب.

✅ هذه هي طريقة البحث النهائية الموحدة والمعتمدة — جاهزة للبناء معماريًا.

✅ إنشاء الطلب — النسخة النهائية المعمارية المغلقة (Unified Final)

🧱 ١️⃣ الهدف
إنشاء طلب بسرعة.
أقل خطوات ممكنة.
منع الأخطاء.
ربط الطلب مباشرة بنتيجة البحث.
منع التضارب قبل حدوثه.

🔥 ٢️⃣ نقطة البداية (Entry Point)
القاعدة الأساسية:
👉 الطلب يبدأ من البحث فقط.
لا يوجد زر "طلب جديد" مستقل خارج سياق البحث.
بيانات البحث تصبح مصدر بيانات الطلب.

👣 ٣️⃣ خطوات إنشاء الطلب

1️⃣ البحث عن القطعة
يمكن عبر:
اسم القطعة.
موديل السيارة.
وصف عام.
رقم تعريف المركبة (VIN).

2️⃣ اختيار القطعة
فتح صفحة القطعة التقنية.
عرض الموردين المرتبطين.

3️⃣ اختيار المورد (للطلب المباشر)
اختيار السعر.
تحديد الكمية.

4️⃣ تأكيد الطلب
عند التأكيد:
إنشاء سجل الطلب في السيرفر.
تسجيل Event creation.
الحالة تصبح:
👉 بانتظار رد المورد.

🔁 ٤️⃣ أنواع الطلب

✅ أ — طلب مباشر
التاجر يختار مورد محدد.
الطلب يرسل مباشرة للمورد.

✅ ب — طلب نشر عام
يستخدم عند:
عدم وجود مورد.
رغبة بعروض متعددة.
النتيجة:
الموردون يقدمون عروض.
التاجر يختار عرض واحد.

🔒 ٥️⃣ قواعد الاتساق أثناء الإنشاء
السيرفر هو مصدر الحقيقة.
إنشاء الطلب عملية ذرية (Atomic).
لا يتم حجز الكمية قبل القبول.
يتم حفظ السعر المعروض كمرجع.
السعر يقفل فقط عند قبول المورد.
لكل طلب معرف idempotency لمنع التكرار.

✏️ ٦️⃣ تعديل الطلب
مسموح:
فقط أثناء حالة "بانتظار رد المورد".
غير مسموح:
بعد القبول.
الحل:
👉 إلغاء وإنشاء طلب جديد.

📡 ٧️⃣ الإشعارات
عند إنشاء الطلب:
إرسال إشعار للمورد (أو الموردين في النشر العام).
تحديث واجهة التاجر فورياً.
الاعتماد على حالة السيرفر وليس ترتيب الإشعارات.

⚠️ ٨️⃣ حالات الفشل
لا يوجد مورد → عرض إنشاء نشر عام تلقائياً.
انقطاع الاتصال → إعادة المحاولة باستخدام idempotency key.
تغير حالة الطلب أثناء العملية → رفض العملية.

🧠 ٩️⃣ قواعد تجربة الاستخدام (UX Rules)
أقل عدد نقرات ممكن.
لا إدخال بيانات يدوياً قدر الإمكان.
استخدام بيانات البحث تلقائياً.
عدم وجود طريق مسدود.

🔥 🔐 1️⃣0️⃣ تحسينات معمارية أساسية (تم التأكد من عدم تضاربها)
هذه ليست ميزات جديدة — بل تثبيتات ضرورية لمنع فوضى الطلبات:
✅ التحقق من الحالة قبل كل عملية (Optimistic locking).
✅ منع تنفيذ نفس الطلب مرتين (Idempotency).
✅ تثبيت snapshot للسعر عند الإنشاء للعرض فقط.
✅ لا يتم تعديل الطلب بعد القبول نهائياً.
✅ العمليات الحساسة تتم في السيرفر فقط.

✅ قواعد الإشعارات النهائية الموحدة 

🧱 ١️⃣ الهدف الأساسي
تنبيه المستخدم فقط عند الأحداث التي تحتاج رد.
تسريع العمل داخل السوق.
منع الإزعاج أو الإشعارات الزائدة.

🔔 ٢️⃣ أنواع الإشعارات (قرار ثابت)
يوجد نوعان فقط:
✅ أ — إشعارات خارج التطبيق
تصل حتى لو التطبيق مغلق.
تستخدم فقط للأحداث المهمة التي تحتاج إجراء.

✅ ب — إشعارات داخل التطبيق
تظهر أثناء استخدام التطبيق.
لا ترسل تنبيه خارجي.

📡 ٣️⃣ الأحداث التي ترسل إشعار خارجي
للمورد:
وصول طلب مباشر جديد.
وصول طلب نشر عام.

للتاجر / الورشة:
قبول الطلب.
جاهزية الطلب للاستلام.
إنشاء دين جديد.
تسجيل دفعة.
طلب نقل ملكية.

📲 ٤️⃣ الأحداث داخل التطبيق فقط
تغييرات حالة الطلب العادية.
تحديثات المخزون.
تعديل بيانات الكتالوج.
تغييرات غير عاجلة.

🧠 ٥️⃣ قاعدة اتخاذ القرار (الأهم)
قبل إرسال إشعار خارجي نسأل:
👉 هل يحتاج المستخدم اتخاذ إجراء الآن؟
إذا نعم → إشعار خارجي.
إذا لا → داخل التطبيق فقط.

🚫 ٦️⃣ منع الإزعاج
لا إرسال إشعار لكل تغيير صغير.
دمج عدة أحداث في إشعار واحد عند الحاجة.

⚡ ٧️⃣ التحديث الفوري داخل التطبيق
عند فتح التطبيق يتم تحديث الأحداث مباشرة.
لا حاجة لإشعار خارجي إضافي.

📥 ٨️⃣ مركز الإشعارات
قائمة داخل التطبيق لكل الأحداث.
تشمل الإشعارات الخارجية والداخلية.

🧩 قواعد تصميم مهمة
الإشعار يعرض الفعل المطلوب وليس فقط معلومة.
نص الإشعار واضح ومباشر.
عدم تكرار نفس الإشعار.

✅ قواعد عرض البيانات النهائية الموحدة 

🧱 ١️⃣ الهدف
عرض البيانات بسرعة عالية.
تقليل التفكير لدى المستخدم.
دعم اتخاذ القرار السريع.
الحفاظ على عدالة السوق.

🔥 ٢️⃣ عرض نتائج البحث (Search Display)
القاعدة الأساسية:
👉 عرض القطعة أولاً وليس المورد.
كل نتيجة تحتوي:
اسم القطعة.
صورة مرجعية (إن وجدت).
السيارات المتوافقة.
مؤشر توفر عام.
لا يتم عرض:
❌ قائمة الموردين كاملة هنا.

👥 ٣️⃣ عرض الموردين داخل القطعة
عند فتح القطعة:
لكل مورد يظهر:
السعر.
حالة التوفر.
مؤشر سرعة الرد (ضمني).
ترتيب العرض حسب نظام الترتيب.

📊 ٤️⃣ نظام ترتيب الموردين (Ranking Logic)
الترتيب يعتمد على:
التوفر الحقيقي.
سرعة الرد.
نجاح العمليات.
النشاط.
الاشتراك = 25%.
قاعدة العدالة:
👉 الاشتراك يعطي أفضلية داخل نفس مستوى الجودة فقط.

🔒 ٥️⃣ رؤية البيانات حسب الدور
المورد:
يرى أسعاره فقط.
لا يرى بيانات الموردين الداخلية.
يمكنه رؤية الأسعار كمشتري عند البحث.

التاجر:
يرى الأسعار لجميع الموردين.
لا يرى تفاصيل مخزون المورد.

الميكانيكي:
يرى السعر والتوفر فقط.

💰 ٦️⃣ عرض السعر
السعر يظهر مباشرة.
السعر يثبت بعد قبول الطلب.
لا يتغير للطلبات القائمة.

📦 ٧️⃣ عرض الكتالوج
القطعة التقنية تظهر مرة واحدة فقط.
منع التكرار.

🔔 ٨️⃣ عرض حالات الطلب
تظهر بوضوح:
بانتظار الرد.
مقبول.
قيد التجهيز.
جاهز للاستلام.
مكتمل.

⚡ ٩️⃣ تحسينات سرعة العرض (Performance Display)
عرض تدريجي للبيانات:
1️⃣ نتائج البحث → بيانات خفيفة فقط.
2️⃣ فتح القطعة → تحميل الموردين.
3️⃣ اختيار المورد → تحميل التفاصيل.
تحميل عند الحاجة فقط:
لا تحميل بيانات ثقيلة مسبقاً.
حفظ البيانات الأخيرة:
القطع المفتوحة مؤخراً تظهر أسرع.

🧠 🔟 قواعد تقليل الارتباك
معلومات أساسية فقط.
لا عرض بيانات تقنية غير ضرورية.
ترتيب واضح وثابت.

✅ تجربة الاستخدام النهائية الموحدة 

🧱 ١️⃣ نقطة البداية (فتح التطبيق)
التاجر / الورشة
الشاشة الرئيسية تحتوي فقط على:
حقل بحث كبير وواضح.
الطلبات الحالية.
الديون النشطة.
آخر الموردين.
القاعدة:
👉 البحث هو مركز النظام وبداية كل عملية.

المورد
الشاشة الرئيسية:
الطلبات الجديدة.
طلبات النشر العام.
الطلبات قيد التجهيز.
ملخص النشاط.

🔎 ٢️⃣ البحث (الدخول الأساسي للنظام)
يوجد طريقتان:
أ — البحث اليدوي
يمكن إدخال:
اسم قطعة.
موديل سيارة.
سنة.
وصف.
رقم قطعة.
كل معلومة إضافية تضيق النتائج.

ب — البحث عبر رقم تعريف المركبة (VIN)
عند إدخال الرقم:
1️⃣ البحث داخلياً أولاً.
2️⃣ استخدام خدمة خارجية فقط عند الحاجة.
3️⃣ تعبئة بيانات السيارة تلقائياً.
4️⃣ حفظ البيانات داخل النظام.

📦 ٣️⃣ عرض النتائج
القرار النهائي:
👉 عرض القطعة أولاً وليس المورد.
داخل القطعة:
قائمة الموردين.
الأسعار.
التوفر.

🔥 ٤️⃣ إنشاء الطلب
لا يوجد زر منفصل.
القاعدة:
👉 البحث = بداية الطلب.
الخطوات:
اختيار المورد.
تحديد الكمية.
إرسال الطلب.

🔁 ٥️⃣ أنواع الطلب
يوجد خياران:
طلب مباشر.
طلب نشر عام.
إذا لا توجد نتائج:
👉 يظهر زر إنشاء طلب نشر عام مباشرة.

👥 ٦️⃣ تجربة المورد
عند وصول الطلب:
يرى التفاصيل.
يمكن قبول أو رفض.
في طلب النشر العام يمكن تقديم عرض.

🔒 ٧️⃣ بعد قبول الطلب
يتم تثبيت السعر.
ينتقل الطلب إلى التجهيز.

📲 ٨️⃣ الاستلام
فتح الطلب.
مسح رمز الاستجابة (QR).
اختيار:
إغلاق نقدي.
تسجيل دين.

💰 ٩️⃣ إدارة الديون
تظهر في قائمة الديون.
يمكن تسجيل دفعات جزئية.

🔁 🔟 نقل الملكية
اختيار قطعة.
إرسال طلب نقل.
قبول الطرف الثاني.
نقل الملكية.

🧠 قواعد تجربة الاستخدام الأساسية
أقل عدد خطوات ممكن.
لا إدخال بيانات معقد.
البحث هو المدخل الأساسي.
لا طريق مسدود داخل التطبيق.
النظام يقود المستخدم

✅ الهيكل المعماري الذكي لمنع تضارب الحالات —  (Final Unified)

🧱 الفكرة الأساسية
👉 الحالة ليست قيمة واحدة ثابتة.
النظام يعمل عبر ثلاث طبقات مترابطة:
1️⃣ الأحداث (Events)
2️⃣ الحالة الحالية (Current State)
3️⃣ قواعد الانتقال (Transition Rules)

🔥 ١️⃣ طبقة الأحداث (Event Layer)
كل عملية داخل النظام تسجل كـ حدث.
لا يتم تغيير الحالة مباشرة بدون تسجيل حدث.
أمثلة أحداث:
إنشاء طلب.
قبول المورد.
محاولة إلغاء.
بدء إجراء مالي.
تسجيل دفعة.
الهدف:
عدم فقدان أي عملية.
إمكانية تتبع ما حدث بدقة.
إمكانية إعادة بناء الحالة من الأحداث.

⚙️ ٢️⃣ فصل الحالات إلى مجالات مستقلة (State Separation)
لا يوجد حقل واحد فقط باسم "الحالة".
الحالات تقسم إلى مجالات:

حالة دورة الطلب:
جديد
بانتظار رد المورد
مقبول
قيد التجهيز
جاهز للاستلام
مكتمل
ملغي
مرفوض

الحالة المالية:
بدون دفع
دين مسجل
دفعات جزئية
مغلق مالياً

حالة التسليم:
غير جاهز
جاهز
تم الاستلام

حالة الملكية:
ملك التاجر
نقل قيد الانتظار
تم النقل

لماذا هذا مهم؟
يمنع تضارب العمليات.
يسمح بعمليات متوازية بدون تعارض.
مثال:
👉 الطلب مقبول + غير مدفوع + غير جاهز.

🔄 ٣️⃣ قواعد الانتقال (State Guard / Transition Rules)
كل تغيير حالة يجب أن يمر عبر قواعد تحقق.
أمثلة:
لا يمكن الاستلام قبل الجاهزية.
لا يمكن نقل الملكية قبل القبول.
لا يمكن تعديل الطلب بعد القبول.
الدور:
👉 منع الحالات غير المنطقية.

⚡ كيفية عمل النظام عملياً
عند حدوث عملية:
1️⃣ يصل الحدث للسيرفر.
2️⃣ التحقق من قواعد الانتقال.
3️⃣ إذا صحيح:
تسجيل الحدث.
تحديث الحالة المرتبطة فقط.

🔥 سر القوة الحقيقي
النظام يعتمد على:
👉 آخر حدث صحيح.
وليس فقط قراءة قيمة الحالة الحالية.
النتيجة:
مقاومة الإشعارات المتأخرة.
مقاومة العمليات المتزامنة.
منع انهيار الحالات.

🚨 لماذا هذا مناسب لأنظمة كبيرة
لأن:
العمليات تحدث بالتوازي.
الاتصال غير مستقر.
المستخدم قد ينفذ نفس العملية أكثر من مرة.
ترتيب الأحداث قد يختلف عن ترتيب وصولها.
هذا الهيكل:
👉 يمنع التضارب بدون تعقيد زائد.

⭐ القاعدة الذهبية
الحالة ليست قيمة ثابتة…
بل:
👉 نتيجة سلسلة أحداث + قواعد انتقال.
